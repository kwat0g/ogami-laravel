  ---
  Recruitment Module — Gap Analysis & Risk Report

  Architecture Overview

  The module spans 8 sub-controllers, 9 services, 13 models, 14 migrations, 7
  policies, 13 notifications, and 27 frontend hooks. The lifecycle is
  well-structured: Requisition → Posting → Application → Interview → Offer → 
  Pre-Employment → Hiring (VP approval) → Employee created.

  ---
  Critical Bugs (will break at runtime)

  1. DB CHECK constraint missing hired status

  File: database/migrations/2026_03_29_000005_create_applications_table.php

  The PostgreSQL CHECK on applications.status lists:
  'new','under_review','shortlisted','rejected','withdrawn' — 'hired' is absent.
   HiringService::vpApprove() sets $application->status = 
  ApplicationStatus::Hired then saves — this will throw a DB constraint
  violation the first time a VP approves a hiring. Fix: add a migration to
  update the constraint.

  2. Missing use Illuminate\Http\Request import in HiringController

  File: app/Http/Controllers/HR/Recruitment/HiringController.php

  vpApprove(Request $request, Hiring $hiring) — if Request isn't imported, PHP
  resolves from global namespace and fails. Needs verification.

  ---
  High Severity — Authorization Gaps

  3. OfferController::store() has no authorization check

  File: app/Http/Controllers/HR/Recruitment/OfferController.php

  The store() method does not call $this->authorize('create', JobOffer::class).
  Any authenticated user with module_access:hr can create an offer for any
  application.

  4. InterviewController::submitEvaluation() has no authorization check

  File: app/Http/Controllers/HR/Recruitment/InterviewController.php

  No authorize('evaluate', $interview) call — any HR module user can submit
  evaluations for interviews they were never assigned to.

  5. No HTTP route for RequisitionService::open()

  File: routes/api/v1/recruitment.php

  After approval, a requisition needs to transition approved → open before
  postings can be properly linked. The open() method exists in the service but
  has no route. Requisitions are stuck at approved.

  ---
  Medium Severity — State Machine Issues

  6. ApplicationStateMachine doesn't include shortlisted → hired transition

  File: app/Domains/HR/Recruitment/Enums/ApplicationStatus.php

  HiringService bypasses the state machine by directly assigning
  $application->status = ApplicationStatus::Hired. This works only because it
  avoids the machine — but it's fragile and means the machine is wrong. The
  machine should formally declare shortlisted → hired.

  7. HiringStatus has no state machine class

  Transitions are ad-hoc if checks in HiringService. The pending status is
  vestigial — created and immediately replaced by pending_vp_approval in the
  same transaction.

  8. OfferService::expireOffer() is unreachable

  No scheduled Artisan command or queued job calls this. Offers past their
  expires_at will never auto-expire — they stay in sent status forever.

  Low Severity — Other Gaps

  11. RecruitmentSeeder not wired into DatabaseSeeder

  php artisan migrate:fresh --seed never creates recruitment demo data. The
  seeder also creates hirings with status = 'hired' and employee_id = null,

  12. Missing SoD on offer creation vs. sending

  No policy rule prevents the same user who prepared an offer from also sending
  it. For high-value positions, a second pair of eyes on the offer before
  sending is good practice.

  13. Interview evaluator can also be the application reviewer

  InterviewPolicy::evaluate() only checks interviewer assignment — no SoD
  prevents the same person who shortlisted an application from also submitting
  the interview evaluation.

  14. Test coverage is minimal

  File: tests/Feature/Recruitment/RecruitmentApiTest.php

  Only 13 test cases. Not covered: approve/reject flows, SoD enforcement,
  posting publish/close/reopen, interview full lifecycle, offer lifecycle,
  pre-employment checklist, VP hiring approval, candidate CRUD, notifications,
  offer expiry, duplicate application guard, department scoping.

  ---
  SoD Exception — HR Manager Creating Their Own Requisition

  Currently RequisitionPolicy::approve() blocks all self-approvals regardless of
   role. Your requirement: HR Managers can approve their own requisitions.

  This is a legitimate business rule (the HR department controls headcount for
  the entire organization, so an HR Manager is also an "organizational
  authority" over their own department's hiring needs). Here is the recommended
  implementation:

  RequisitionPolicy.php — change the approve() method:

  public function approve(User $user, JobRequisition $requisition): bool
  {
      // HR Manager exception: may approve their own requisitions
      if ($user->hasPermissionTo('recruitment.requisitions.approve')
          && $user->hasRole(['manager'])
          && $user->employee?->department?->code === 'HR') {
          return true;
      }

      // Standard SoD: cannot approve what you created
      if ($user->id === $requisition->requested_by) {
          return false;
      }

      return $user->hasPermissionTo('recruitment.requisitions.approve');
  }

  RequisitionService.php — update the SoD guard at service layer:

  // Inside approve()
  $isHrManager = $actor->hasRole('manager')
      && $actor->employee?->department?->code === 'HR';

  if (!$isHrManager && $actor->id === $requisition->requested_by) {
      throw new DomainException(
          'You cannot approve your own requisition.',
          'SOD_SELF_APPROVAL',
          403
      );
  }
