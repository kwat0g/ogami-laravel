## 1. HR

**What it manages:** Employee master data, departments, positions, and salary grades.

**Process:**
1. HR creates an employee record (`draft`).
2. Documents are collected → status moves to `active` (onboarding complete).
3. Employee may go `on_leave` or `suspended` (temporary), then return to `active`.
4. Separation ends in `resigned` or `terminated` (terminal).

**States:** `draft → active ↔ on_leave | suspended → resigned | terminated`

**Key rules:**
- `daily_rate` and `hourly_rate` are PostgreSQL-generated columns (never set manually).
- Government IDs stored encrypted + SHA-256 hash for uniqueness checks.
- Department scoping: employees visible only to their department and above (except admin/executive).

**Actors:** HR Manager, Department Head, Employee
**Feeds into:** Payroll, Attendance, Leave, Loan, Procurement (budget/approval)

## 2. Payroll

**What it manages:** Semi-monthly payroll computation and multi-level approval for all employees.

**Process:**
1. Payroll Admin creates a run (`DRAFT`) and sets scope (departments, positions, employment types).
2. Pre-run checks validate attendance data, loan amortizations, leave balances (`PRE_RUN_CHECKED`).
3. System computes pay via a 17-step pipeline (`PROCESSING → COMPUTED`).
4. HR reviews computed results (`REVIEW → SUBMITTED → HR_APPROVED`).
5. Accounting approves (`ACCTG_APPROVED`) → VP signs off (`VP_APPROVED`).
6. Disbursement processed; GL entries auto-posted (`DISBURSED → PUBLISHED`).
7. Any approver can return the run (`RETURNED → DRAFT`) for corrections.

**States:**
```
DRAFT → SCOPE_SET → PRE_RUN_CHECKED → PROCESSING → COMPUTED → REVIEW →
SUBMITTED → HR_APPROVED → ACCTG_APPROVED → VP_APPROVED → DISBURSED → PUBLISHED
(RETURNED / REJECTED → DRAFT)
```

**17-Step Pipeline:**

| Step | What it computes |
|------|-----------------|
| 01 Snapshots | Capture employee rates & benefits at run time |
| 02 Period Meta | Pay period dates, working-day count |
| 03 Attendance | Worked days, absences, late/undertime |
| 04 YTD | Year-to-date totals |
| 05 Basic Pay | Base salary |
| 06 Overtime | OT pay (1.25× regular, 1.30× rest day, 1.50× holiday) |
| 07 Holiday | Regular/special holiday pay |
| 08 Night Diff | Night differential (10% for 10 PM–6 AM) |
| 09 Gross Pay | Sum of all earnings |
| 10 SSS | SSS contribution (employee + employer share) |
| 11 PhilHealth | PhilHealth premium |
| 12 Pag-IBIG | Pag-IBIG contribution |
| 13 Taxable | Taxable income (gross − statutory exemptions) |
| 14 WHT | Withholding tax (TRAIN law brackets) |
| 15 Loan Deductions | Monthly loan amortizations |
| 16 Other Deductions | Insurance, cash advances, uniform, others |
| 17 Net Pay | Gross − all deductions |

**Key rules:**
- SoD: initiator ≠ each approver.
- Returning a run clears scope and all exclusions for a clean restart.
- Batch computation is queued; progress tracked in real time.

**Actors:** Payroll Admin, HR Manager, Accounting Manager, VP
**Feeds into:** Accounting (GL posting), Tax (WHT/EWT reporting)

## 3. Accounting

**What it manages:** General ledger, journal entries, bank reconciliation, and financial reports.

**Process:**
1. Entries arrive as `draft` (manually created or auto-generated by Payroll, AP, AR, Fixed Assets).
2. Clerk submits → Manager posts → status becomes `posted`.
3. Posted entries update GL balances; fiscal period must be open.
4. Monthly: bank transactions imported → reconciliation matches bank ↔ GL (`pending → reconciled`).
5. Reports generated: Balance Sheet, Income Statement, Trial Balance, Cash Flow.

**States (Journal Entry):** `draft → submitted → posted | cancelled`
**States (Bank Recon):** `pending → in_progress → reconciled | exception`

**Key rules:**
- Double-entry enforced: debit total must equal credit total on every JE.
- JE number (JE-YYYY-MM-NNNNNN) assigned only on posting.
- Fiscal period enforcement: date must fall within an open period.
- Reversal support: a JE can reverse another for corrections.

**Actors:** Accounting Clerk, Accounting Manager, Finance Controller
**Feeds into:** Tax (reporting basis), Budget (actuals), Fixed Assets (depreciation GL)

## 4. AP — Accounts Payable

**What it manages:** Vendor invoices, payments, credit notes, and the purchase-to-pay cycle.

**Process:**
1. AP Clerk creates invoice linked to a Purchase Order and Goods Receipt (`draft`).
2. 3-Way Match verified: PO ↔ GR ↔ Invoice quantities and amounts must align.
3. Multi-level approval: Department Head notes → Manager checks → Officer reviews → `approved`.
4. Payment scheduled → processed → invoice status becomes `paid` or `partially_paid`.
5. GL entry auto-posted on approval (debit Expense/COGS, credit AP Liability).

**States:** `draft → pending_approval → head_noted → manager_checked → officer_reviewed → approved → partially_paid | paid`

**Key rules:**
- EWT rate and amount frozen at invoice creation (prevents rate drift).
- Net payable = net amount + VAT − EWT.
- SoD: each approver must be a different person.
- VAT exemption reason required if no VAT.

**Actors:** AP Clerk, Dept Head, Manager, Accounting Officer, Finance Controller
**Feeds into:** Accounting (GL), Tax (EWT/VAT reporting), Cash management

---

## 5. AR — Accounts Receivable

**What it manages:** Customer invoices, payments, credit notes, and the order-to-cash cycle.

**Process:**
1. AR Clerk creates invoice linked to a Delivery Receipt (`draft`).
2. Manager approves → invoice number assigned (INV-YYYY-MM-NNNNNN) → status `approved`.
3. Customer makes full or partial payment → status `paid` or `partially_paid`.
4. Bad debts: VP approves write-off → reversal GL entry posted.

**States:** `draft → approved → partially_paid → paid | written_off | cancelled`

**Key rules:**
- Delivery Receipt must exist before invoice can be created (ensures goods were actually delivered).
- Invoice number assigned only on approval, not on creation.
- Write-off requires VP approval + GL reversal.

**Actors:** Sales Clerk, Sales Manager, Finance Controller, VP
**Feeds into:** Accounting (GL), Tax (output VAT), Cash management

## 6. Tax

**What it manages:** Philippine BIR tax filing tracking and VAT ledger reconciliation.

**Process:**
1. System aggregates tax data from Payroll (WHT), AP (EWT, input VAT), and AR (output VAT).
2. Tax officer creates a BIR filing record for the relevant form and period.
3. Filed on time → `filed`; filed late → `late`; amendment submitted → `amended`.
4. VAT Ledger reconciles input VAT (from AP) vs. output VAT (from AR) for each period.

**States (BIR Filing):** `pending → filed | late | amended | cancelled`

**BIR Forms covered:**

| Form | Description |
|------|-------------|
| 1601C | Monthly WHT on compensation (from Payroll) |
| 0619E | Monthly EWT remittance (from AP) |
| 1601EQ | Quarterly EWT return |
| 2550M | Monthly VAT declaration |
| 2550Q | Quarterly VAT return |
| 1702Q | Quarterly income tax |
| 1702RT | Annual income tax (RCIT) |
| 2307 alpha | Annual alphalist of EWT |

**Actors:** Accounting Manager, Tax Specialist, Finance Controller
**Feeds into:** Government compliance filing

## 7. Procurement

**What it manages:** Purchase requests, purchase orders, goods receipts, and vendor RFQs.

**Process:**
1. Department creates Purchase Request (`draft`) — budget checked against annual allocation.
2. Purchasing reviews validity → Accounting verifies budget → VP approves.
3. On approval, Purchase Order auto-created and sent to vendor.
4. Vendor acknowledges → delivers → Goods Receipt created in the system.
5. GR feeds into 3-Way Match for AP invoice verification.

**States (Purchase Request):**
```
draft → pending_review → reviewed → budget_verified → approved → converted_to_po
(returned | rejected → draft)
```
**States (Purchase Order):** `draft → submitted → acknowledged → partially_received → received_in_full | cancelled`
**States (Goods Receipt):** `draft → confirmed | rejected`

**Key rules:**
- PR creation hard-blocked if department annual budget is exceeded.
- SoD: requester ≠ reviewer ≠ budget verifier ≠ VP approver.
- RFQ module allows comparing vendor quotes before committing to a PO.

**Actors:** Dept Manager, Purchasing Agent, Accounting Manager, VP
**Feeds into:** Inventory (stock receipt), AP (invoice 3-way match), Budget (consumption)

## 8. Inventory

**What it manages:** Item master, stock balances, stock movements, and material requisitions.

**Process:**
1. Goods received from Procurement (GR) → stock ledger entry `goods_receipt` → balance increases.
2. Production opens Material Requisition → picker picks items from location → `issued` → balance decreases.
3. Production outputs finished goods → `production_output` → balance increases.
4. Adjustments (cycle counts, scraps) posted as `adjustment` entries.
5. Physical count done periodically → variance reconciled.

**States (Material Requisition):** `draft → submitted → picked → issued | cancelled`

**Key rules:**
- Stock balance is a denormalized running total updated on every ledger post.
- Lot/batch tracked for traceability and FIFO costing.
- Stock reservations block inventory for open production orders.
- Low-stock alert event fired when balance drops below reorder point.

**Actors:** Warehouse Manager, Receiving Clerk, Production Planner, Inventory Auditor
**Feeds into:** Production (material issuance), Accounting (COGS valuation)

## 9. Production

**What it manages:** Production orders, bill of materials, delivery schedules, and output tracking.

**Process:**
1. Client order or delivery schedule triggers a Production Order (`draft`).
2. BOM attached → Material Requisition auto-created for required components.
3. Production starts (`in_progress`); output logged per batch (qty produced, qty rejected).
4. QC inspection validates output.
5. Finished goods received into Inventory → PO marked `completed`.

**States:** `draft → scheduled → in_progress → on_hold → completed | cancelled`

**Key rules:**
- BOM defines components + quantities + scrap allowance.
- Progress % = qty produced / qty required.
- On-hold reason tracked (material shortage, machine down, etc.).
- Actual vs. target date tracked for delivery commitment visibility.

**Actors:** Production Manager, Production Planner, Shop Floor Supervisor, QC Inspector
**Feeds into:** Inventory (finished goods receipt), QC (inspection), AR (delivery link)

## 10. QC — Quality Control

**What it manages:** Inspection checklists, inspection records, and non-conformance reports (NCR).

**Process:**
1. Inspection triggered on GR receipt (incoming) or production output (in-process/final).
2. Inspector fills out checklist based on assigned template.
3. Result: `passed` (released), `failed_hold` (hold for decision), `reworked`, or `scrap`.
4. If failed: NCR raised → root cause investigated → corrective action assigned → `closed`.

**States (Inspection):** `draft → in_progress → passed | failed_hold | reworked | scrap`
**States (NCR):** `raised → under_investigation → corrective_action_assigned → closed | reopened`

**Key rules:**
- Inspection template is reusable per stage (receiving, in-progress, final).
- NCR tracks root cause analysis and corrective/preventive action (CAPA).
- Failed incoming GR can trigger return-to-vendor or scrap disposition.

**Actors:** QC Inspector, QC Manager, Production Manager
**Feeds into:** Inventory (gates stock receipt), Production (rework or scrap decision)

## 11. Maintenance

**What it manages:** Equipment register, work orders, preventive maintenance schedules, and spare parts.

**Process:**
1. Equipment breakdown reported → Work Order created (`reported`), type: `corrective`.
2. Scheduled PM triggers a new WO automatically, type: `preventive`.
3. WO assigned to technician → work begins (`in_progress`) → spare parts issued from Inventory.
4. Repair complete → WO closed (`completed`); downtime duration recorded.

**States (Work Order):** `reported → assigned → in_progress → completed | cancelled`

**Key rules:**
- Priority levels: critical, high, normal, low — critical WOs flag production impact.
- Labor hours logged against each WO for cost tracking.
- Preventive maintenance schedule driven by calendar (monthly/quarterly).
- Mold-specific WOs link to mold master for shot-count-triggered maintenance.

**Actors:** Maintenance Supervisor, Technician, Equipment Operator
**Feeds into:** Production (downtime visibility), Inventory (spare parts consumption), Mold (preventive schedule)

## 12. Mold

**What it manages:** Mold master catalog, shot count tracking, and end-of-life (EOL) alerts.

**Process:**
1. Mold registered with max shot limit and cavity count.
2. Each Production Order output increments the mold's shot counter.
3. When shots approach max limit → alert triggered → Preventive Maintenance WO auto-created.
4. Mold sent for maintenance → status `under_maintenance` → returns to `active` or `retired`.

**States:** `active → under_maintenance | retired | inactive`

**Key rules:**
- Cavity count directly affects production cycle output (units per shot).
- Shot count is the primary EOL indicator.
- Last maintenance date tracked for PM scheduling.

**Actors:** Production Manager, Maintenance Supervisor, Mold Engineer
**Feeds into:** Maintenance (preventive WO trigger), Production (availability)

## 13. Delivery

**What it manages:** Delivery receipts, shipments, fleet (vehicles), and import/export documents.

**Process:**
1. Sales prepares Delivery Receipt (`draft`) linked to customer invoice items.
2. Goods packed → DR moves to `ready_for_pickup`.
3. Vehicle assigned → shipment dispatched (`in_transit`).
4. Customer confirms receipt → DR becomes `delivered`.
5. Customer signature/confirmation feeds back to AR as proof of delivery.

**States (Delivery Receipt):** `draft → ready_for_pickup → in_transit → delivered | returned | cancelled`
**States (Shipment):** `pending → shipped → in_transit → delivered | lost | returned`

**Key rules:**
- Delivery Receipt must exist before AR invoice can be created (physical-delivery gate).
- Fleet module tracks vehicle plate, capacity, and trip assignments.
- ImpEx documents (air waybill, bill of lading) attached for export shipments.

**Actors:** Sales Admin, Logistics Manager, Delivery Driver, Customer
**Feeds into:** AR (invoicing gate), Inventory (stock deduction on delivery)

## 14. ISO

**What it manages:** QMS document control, internal audits, audit findings, and corrective actions (CAPA).

**Process:**
1. Document drafted → reviewed by owner/manager → `effective` (replaces previous version).
2. Annual/periodic internal audit scheduled → auditor executes → findings logged.
3. Each finding tracked: root cause investigated → corrective action assigned → evidence collected → `closed`.
4. CAPA follow-up: improvement action monitored until verified complete.

**States (Document):** `draft → under_review → effective | superseded | archived`
**States (Audit):** `scheduled → in_progress → completed | cancelled`
**States (Finding):** `raised → under_investigation → closed`

**Key rules:**
- Document versioning: each revision logged with effective date and change summary.
- Superseded versions archived (not deleted) for traceability.
- Finding closure requires evidence of corrective action.

**Actors:** QMS Manager, Internal Auditor, Department Owner, Executive
**Feeds into:** Production (process procedures), QC (inspection standards), HR (training records)

## 15. CRM

**What it manages:** Client orders (portal), negotiation rounds, delivery schedules, and support tickets.

**Process:**
1. Customer submits order via Client Portal.
2. Sales reviews → negotiation round begins (Sales counter-proposes price/terms).
3. Multi-round negotiation until both sides agree → `approved`.
4. Approved order triggers Production Order + Delivery Schedule.
5. Customer raises support tickets for issues; team responds and closes.

**States (Client Order):** `pending → negotiating ↔ client_responded → approved | rejected | cancelled`
**States (Ticket):** `open → in_progress → resolved | closed | reopened`

**Key rules:**
- SLA deadline enforced for sales response time.
- Negotiation round counter tracks how many rounds occurred.
- Phased delivery: order may have multiple delivery milestones with payment triggers.

**Actors:** Customer, Sales Rep, Sales Manager, VP
**Feeds into:** Production (schedule), AR (invoice), Delivery (DR creation)

## 16. Fixed Assets

**What it manages:** Asset register, depreciation schedules, and disposals.

**Process:**
1. Asset acquired and registered (code auto-generated by PostgreSQL trigger).
2. Depreciation method selected (straight-line, double-declining, units-of-production).
3. Monthly scheduler auto-posts depreciation Journal Entry → accumulated depreciation grows.
4. Asset reaches `fully_depreciated` → may continue in use at zero book value.
5. Disposal: book value vs. sale price → gain/loss calculated → reversal GL entry posted.

**States:** `active → fully_depreciated | disposed | impaired`

**Depreciation methods:**
- **Straight-line:** (Cost − Residual) ÷ Useful Life ÷ 12 per month
- **Double-declining:** 2 × SL rate on declining book value
- **Units-of-production:** Cost per unit produced × actual output

**Key rules:**
- Asset code is set by a PostgreSQL trigger only — never set in PHP.
- Depreciation JE auto-posted monthly via scheduler (no manual trigger needed).
- Disposal posts: debit Accumulated Depreciation + debit/credit Gain/Loss, credit Asset.

**Actors:** Asset Manager, Finance Controller, Accountant
**Feeds into:** Accounting (monthly depreciation JE, disposal GL)

## 17. Budget

**What it manages:** Annual departmental budgets by GL account and cost center.

**Process:**
1. Department manager drafts budget per GL account for the fiscal year.
2. Finance Controller reviews and approves (`approved`).
3. Throughout the year, actual GL spend compared against budget.
4. Procurement PRs hard-blocked if department budget would be exceeded.

**States:** `draft → submitted → approved | rejected`

**Key rules:**
- Budget stored in centavos (integer) per department + GL account + fiscal year.
- Budget vs. actual is a live query against GL balances.
- Budget check on PR is a hard block (not just a warning).
- Cost centers allow sub-department budget tracking.

**Actors:** Department Manager, Finance Controller, Budget Analyst
**Feeds into:** Procurement (PR budget gate), Accounting (budget vs. actual reports)

## 18. Attendance

**What it manages:** Daily attendance logs, shift assignments, overtime requests, and holiday calendar.

**Process:**
1. Attendance logs imported daily from biometric machine CSV or entered manually.
2. System calculates worked minutes, late/undertime, absences per employee per day.
3. Overtime pre-approved by supervisor before hours worked; OT minutes logged.
4. Holiday calendar marks regular/special non-working/special working days.
5. Attendance data consumed by Payroll on each run.

**States (OT Request):** `pending → approved | rejected`

**Key rules:**
- One attendance log per employee per work date (unique constraint).
- Night differential: 10% premium for hours worked between 10 PM and 6 AM.
- Source tracked: `biometric | csv_import | manual | system`.
- Leave taken on a day sets the `is_absent` flag automatically.

**Actors:** HR Admin, Employee, Supervisor, Payroll Processor
**Feeds into:** Payroll (Steps 3, 6, 8 — basic pay, overtime, night diff)

## 19. Leave

**What it manages:** Leave requests, leave type definitions, and leave balance tracking.

**Process:**
1. Employee files a leave request (`draft → submitted`).
2. Four-step approval chain: Dept Head → Manager → GA Officer → VP.
3. GA Officer sets action: approved with pay, approved without pay, or disapproved.
4. VP final approval triggers balance deduction (only at this step, not before).
5. Leave balance accrues monthly per leave type configuration.

**States:**
```
draft → submitted → head_approved → manager_checked → ga_processed → approved | rejected
```

**Key rules:**
- SoD: each approver must be a different person from the submitter (LV-004).
- Without-pay leaves do not deduct from leave balance.
- Half-day leave supported (AM or PM).
- Balance snapshot recorded at GA processing step: beginning, applied, ending.
- Monetizable leave types (e.g., unused VL) can be paid out at year-end.

**Actors:** Employee, Dept Head, Manager, GA Officer, VP
**Feeds into:** Payroll (leave balance, unpaid-leave day count), Attendance (absent flag)

## 20. Loan

**What it manages:** Employee loan applications, amortization schedules, and monthly payroll deductions.

**Process:**
1. Employee requests a loan → amortization schedule auto-generated.
2. Multi-stage approval: Head notes → Manager checks → Officer reviews → Supervisor approves.
3. Loan approved → `ready_for_disbursement` → funds released → status `active`.
4. Each payroll run deducts the monthly amortization (Step 15 of payroll pipeline).
5. Final payment → `fully_paid`; forgiven loans → `written_off` + GL reversal.

**States:**
```
pending → head_noted → manager_checked → officer_reviewed → supervisor_approved →
approved → ready_for_disbursement → active → fully_paid | written_off
```

**Key rules:**
- SoD: approver ≠ requestor (enforced at DB level).
- Monthly interest = Principal × Annual Rate ÷ 12.
- Payroll deducts only from `active` loans (not pending or disbursement-ready).
- Write-off posts a GL reversal entry.

**Actors:** Employee, Dept Head, Manager, Supervisor, Accounting Manager
**Feeds into:** Payroll (monthly deduction, Step 15), Accounting (disbursement + write-off GL)

## Cross-Module Flows

### Purchase-to-Pay (P2P)
```
Budget check → PR (Procurement) → PO → Vendor delivers → GR (Inventory) →
AP Invoice (3-way match) → AP Approval → GL Posting (Accounting) → Payment
```

### Order-to-Cash (O2C)
```
Client Order (CRM) → Negotiation → Production Order → Material Issuance (Inventory) →
QC Inspection → Finished Goods → Delivery Receipt → AR Invoice → Customer Payment →
GL Posting (Accounting)
```

### Payroll Cycle
```
HR (employee master, rates) + Attendance (worked days/OT) + Leave (days absent) +
Loan (amortizations) → Payroll Pipeline (17 steps) → Disbursement →
GL Journal Entry (Accounting) → WHT Filing (Tax)
```

### Inventory Flow
```
GR Receipt (Procurement) → Stock Ledger +
Material Requisition (Production) → Stock Ledger −
Production Output → Stock Ledger +
Delivery (Sales) → Stock Ledger −
```

## Roles Quick Reference

| Role | Key Access |
|------|-----------|
| `super_admin` / `admin` | System-wide configuration, user management |
| `executive` | Read-only across all modules, VP approvals |
| `vice_president` | Final approval on payroll, leave, loans, procurement |
| `manager` | Domain operations for their department (module access by dept) |
| `officer` | Accounting, GA, purchasing operations |
| `head` | Department head approvals (leave, loan, PR) |
| `staff` | Basic data entry in their department |
| `vendor` | Vendor Portal only (PO fulfillment, catalog, invoices) |
| `client` | Client Portal only (orders, tickets) |

> **Note:** `manager`, `head`, and `staff` are scoped to their department. `executive` and `vice_president` see all departments. `admin` has only `system.*` permissions — not HR/payroll by default.
